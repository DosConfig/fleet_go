# Fleet Go

DRT(Demand-Responsive Transit) 셔틀 배차 관리 앱.
승객 호출, 드라이버 배차, 관제 모니터링 기능을 구현한 Flutter 프로젝트.

실시간 위치 처리, 대량 마커 렌더링 최적화, 배차 동시성 처리, 오프라인 동기화 등
실무에서 다뤄보고 싶었지만 기회가 없었던 기술들을 직접 설계하고 구현해보기 위한 프로젝트입니다.

## 아키텍처

Feature-First + Clean Architecture.

```
lib/
├── core/              # DI, 인프라 (Firebase, drift, 네트워크)
└── features/
    ├── auth/          # Google/Email 인증
    ├── control/       # 관제 (실시간 지도 + 차량 마커)
    ├── driver/        # 드라이버 (배차 수락, 상태 전이)
    ├── location/      # GPS 수집 + RTDB 동기화
    ├── passenger/     # 승객 (호출, 대기, 탑승)
    ├── route/         # TMAP 경로 탐색
    └── trip/          # 배차 상태 머신
```

## 기술 스택

| 영역 | 기술 |
|------|------|
| Framework | Flutter 3.44 / Dart 3.12 |
| 상태관리 | Riverpod 3 (`@riverpod` + code gen) |
| 상태 모델링 | freezed sealed class |
| 인증 | Firebase Auth (Google + Email) |
| DB | Cloud Firestore + Realtime Database + drift |
| 경로 | TMAP REST API (Dio) |
| 지도 | Naver Map SDK |
| 동시성 | Dart Isolate (모의 차량 위치 연산 워커) |

## 성능 측정 — 대량 마커 렌더링

모의 차량 **1,000대를 500ms 주기**로 갱신하며(위치 연산은 UI 스레드 밖 isolate 워커)
세 가지 렌더 전략을 같은 조건에서 비교한다.

| 모드 | 전략 | 갱신 1회당 비용 |
|------|------|------|
| **A. 전체 재생성** | `clearOverlays` 후 마커 전량 생성·재추가 (초기 방식) | 네이티브 오버레이 생성/제거 × 차량 수 |
| **B. 캐시+diff** | `NMarker` 인스턴스 캐시, `setPosition`만 호출 | 위치 갱신 채널 호출 × 차량 수 |
| **C. 컬링+클러스터** | B + viewport 컬링, 화면상 1.5px 미만 이동 스킵, 줌아웃 시 그리드 클러스터 | 화면에 보이는 수에만 종속 |

**측정 방법 — 자동 벤치마크**: debug 빌드로 관제 화면에 진입하면 시나리오가 자동 실행된다.
차량 규모 1,000 → 5,000 → 10,000대를 순회하며 각 규모에서 (B@z14, C@z14, C@z12),
1,000대에서는 A까지 측정한다. 조건마다 카메라를 고정 기준점 프리셋으로 재현하고,
안정화 4초 대기 후 12초(5초 윈도우 2개)씩 수집 — 사람 손이 개입하지 않아 항상 동일 조건.
종료 시 콘솔에 `[BENCH]` 결과 블록이 한 번에 출력된다.

수동 측정도 가능: 하단 컨트롤에서 모드(A/B/C)·줌 프리셋(Z14/Z12) 고정 후 ▶ 버튼을
누른 시점부터만 수집(조건 변경 시 자동 정지 — 이전 조건 프레임 미누적).
각 윈도우 로그는 `[PERF][모드@줌]` 형식으로 avg / p95 / worst / jank(>16.7ms),
build/raster 분해, `[markers=가시/전체]` 컨텍스트를 포함한다.

**결과 1 — 전 차량 화면 내 밀집 조건** (5초 윈도우 = 갱신 틱 10회, 안정 구간):

| 모드 | avg (ms) | p95 (ms) | worst (ms) | jank/윈도우 |
|------|------|------|------|------|
| A. 전체 재생성 | 36~42 | **250~258** | 258 | 9~18 |
| B. 캐시+diff | 11~12 | **14~15** | 15 | **0** |
| C. 컬링+클러스터 | 7~12 | 18~20 | 29 | 1~4 |

- **A → B**: 매 틱 반복되던 ~255ms 프레임 스톨(1,000개 삭제·재생성 비용)이 사라짐. p95 기준 약 17배 개선, jank 0
- 이 조건은 전 차량이 화면 안에 밀집해 **컬링·클러스터가 발동하지 않는** C의 최악 케이스 — B와 동급으로 나온 이유. A/B 비교로는 유효(둘 다 동일한 최악 조건)
- 모드 전환 직후 첫 윈도우(마커 일괄 재생성 과도기)는 제외

**결과 2 — 도시 전역 분산 조건** (차량 1,000대를 ±0.15°, 약 33km 범위로 분산.
측정 시작 직후 안정화 윈도우 제외, 화면 내 개별 마커는 100개 미만):

| 모드 | 조건 | avg (ms) | p95 (ms) | jank/윈도우 |
|------|------|------|------|------|
| A. 전체 재생성 | 줌 14 | 34~55 | **246~250** | 3~7 |
| A. 전체 재생성 | 줌 12 | 31~62 | **239~251** | 2~7 |
| B. 캐시+diff | 줌 14 | 4.6~6.3 | 8.2 | 0 |
| B. 캐시+diff | 줌 12 | 4.4~4.8 | 7.9~8.2 | 0 |
| C. 컬링+클러스터 | 줌 14 (컬링) | 3.1~4.2 | 5.5~15.0 | 0 |
| C. 컬링+클러스터 | 줌 12 (클러스터) | 6.5~8.1 | 14.9~16.3 | 0 |

- **A는 줌과 무관하게 p95 ~250ms** — 화면에 몇 개가 보이든 전체 차량 수만큼 비용을 지불한다는 직접 증거
- **B와 C는 1,000대 규모에선 프레임 시간 동급** — 지도 SDK가 화면 밖 오버레이를 그리지 않으므로, 렌더 비용은 이미 가시 영역에 종속. C가 추가로 줄이는 것은 플랫폼 채널 호출 수와 네이티브 오버레이 객체 수
- 각 윈도우의 `[markers=가시/전체]`, `[clusters=셀/전체]` 컨텍스트가 로그에 기록됨

**결과 3 — 규모 스케일링과 병목 재특정** (1,000 → 100,000대, Float64List 파이프라인):

| 차량 수 | B p95 (ms) | C@z14 p95 (ms) | C 가시 마커 | C@z12 p95 (ms) | 클러스터 셀 |
|------|------|------|------|------|------|
| 1,000 | 17.5~18.1 | 14.4~15.1 | 15 | 27.4~27.6 | 31~32 |
| 5,000 | 5.8~16.2 | 11.6~12.9 | 58 | 25.3~27.0 | 35 |
| 10,000 | 8.9~18.4 | 11.7~13.3 | 118~122 | 23.7~26.9 | 35 |
| 100,000 | (제외) | **10.2~12.8** | 1,120~1,143 | 12.4~18.7 | **35** |

- **병목 재특정**: 초기 파이프라인(isolate가 매 틱 Map 리스트를 직렬화 전송)에서는
  B가 10,000대에서 **프레임 1.19초로 붕괴**했다. 분해 계측상 build/raster는 1~2ms —
  범인은 마커 갱신(setPosition)이 아니라 **isolate 경계의 Map 직렬화**였고,
  전송을 `Float64List`(`[lat, lng, heading, speed] × N`, 버퍼째 저비용 전송)로
  전환하자 같은 조건의 B가 p95 9~18ms로 정상화됐다. 병목은 규모에 따라
  렌더 → 데이터 파이프라인으로 이동하며, 계측 없이는 귀속을 틀리기 쉽다는 사례
- **C는 100,000대에서도 p95 10~13ms, jank 0** — 가시 마커 약 1,130개.
  클러스터 셀은 그리드가 화면 크기에 종속이라 1,000대든 100,000대든 **35개 고정**.
  "갱신 비용을 전체 수가 아닌 화면에 보이는 수에 종속시킨다"는 설계 목표의 직접 증명
- z12(클러스터) 윈도우 간 프레임 시간 변동(12~27ms)은 지도 타일 렌더가 겹친
  시뮬레이터 환경 특성으로, 규모 축과는 무관(100,000대가 1,000대보다 빠르게 측정되기도 함)

> 측정 환경: iPhone 16 Pro, debug 모드. 절대값보다 동일 조건 상대 비교가 목적.

```bash
flutter pub get
dart run build_runner build
flutter run
```

`.env` 파일에 `TMAP_APP_KEY`, `NAVER_MAP_CLIENT_ID` 설정 필요.
