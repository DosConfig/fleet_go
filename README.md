# Fleet Go

DRT(수요응답형 교통) 셔틀 배차 관제 데모입니다. 승객 호출, 드라이버 배차, 관제 모니터링까지의 전체 흐름을 구현했고, 서울 전역에서 10만 대의 차량이 0.5초마다 움직이는 극한 조건에서도 지도 렌더링이 버티는지 실측으로 검증했습니다.

실서비스에서 10만 대 동시 관제 같은 상황은 일어나지 않을 수 있지만, 한계 규모를 미리 실측해두면 설계가 어디까지 버티고 어디서 왜 무너지는지 알고 시작할 수 있다고 생각해서 정상 시나리오보다 훨씬 가혹한 조건까지 실험을 확장했습니다.

## 실행 방법

```bash
flutter pub get
dart run build_runner build
flutter run
```

`.env` 파일에 `TMAP_APP_KEY`, `NAVER_MAP_CLIENT_ID` 설정이 필요합니다.

테스트 환경: Flutter 3.44 / Dart 3.12, iPhone 16 Pro 시뮬레이터

## 프로젝트 구조

Feature-First 클린 아키텍처입니다. 피처 단위로 data/domain/presentation을 묶어 피처 추가가 기존 코드에 영향을 주지 않도록 했습니다.

```
lib
├── core
│   ├── di               # 전역 DI (repository providers)
│   └── perf             # FrameLogger (성능 계측기)
└── features
    ├── auth             # Google/Email 인증
    ├── control          # 관제 (실시간 지도, 렌더 벤치마크)
    ├── driver           # 드라이버 (배차 수락, 상태 전이)
    ├── location         # GPS 수집 + RTDB 동기화
    ├── passenger        # 승객 (호출, 대기, 탑승)
    ├── route            # TMAP 경로 탐색
    ├── sync             # drift 오프라인 동기화 큐
    └── trip             # 배차 상태 머신
```

## 구현 기능 목록

핵심 기능

- 배차 플로우: 승객 호출, 기사 수락, 픽업 이동, 탑승, 운행, 하차 완료, 양측 취소
- 배차 상태 머신: freezed sealed class로 상태와 전이 규칙 정의
- 동시 수락 경합 처리: Firestore 트랜잭션으로 원자적 상태 전이
- 실시간 위치: 드라이버 GPS를 RTDB로 올려 관제/승객 지도에 표시
- 경로 안내: TMAP API 경로 탐색, 폴리라인과 출발/도착 마커 표시
- 배차 복구: 앱 재실행 시 진행 중 배차를 Firestore에서 복구
- 오프라인 대응: drift 기반 동기화 큐, 네트워크 복구 시 자동 동기화

관제 성능 (이 프로젝트의 핵심 실험)

- 실시간 마커 렌더링: 마커 캐시 + diff 갱신, viewport 컬링, 픽셀 미만 이동 스킵, 줌아웃 그리드 클러스터링
- isolate 워커: 차량 위치 연산을 UI 스레드 밖에서 수행, Float64List 버퍼로 전송
- 자동 벤치마크: 규모(1천~10만 대) x 렌더 전략 x 줌 조건을 사람 개입 없이 순회 측정

## 앱 구조 설계

상태 관리 방식 선택 이유

Riverpod 3.0+ (@riverpod annotation)을 선택했습니다. 배차 상태는 승객과 기사 두 화면이 동시에 봐야 하므로, 메모리 상태와 서버 상태가 이원화되면 반드시 어긋납니다. 초기에 메모리 프로바이더와 Firestore Stream을 병행했다가 동기화 버그가 반복되어, Firestore Stream을 단일 소스로 두고 화면은 그것만 watch하는 구조로 정리했습니다.

배차 상태 머신 설계

배차 상태를 freezed sealed class로 정의하고 전이 규칙을 코드로 강제했습니다. 여러 기사가 같은 콜을 동시에 수락하는 경합은 클라이언트에서 막을 수 없어서, 상태 전이를 Firestore 트랜잭션 안에서 검증 후 커밋하는 방식으로 처리했습니다.

위치 연산 분리

모의 차량의 위치 갱신 연산은 isolate 워커에서 수행합니다. UI 스레드는 수신한 위치를 그리기만 합니다. isolate 경계의 전송 프로토콜은 아래 성능 측정 과정에서 Map 리스트에서 Float64List 버퍼로 교체했습니다.

## 성능 측정

측정 방법

- 서울 전역(약 33km) 범위에 분산된 모의 차량 전체가 500ms 주기로 위치 갱신
- FrameLogger가 5초 윈도우 단위로 avg / p95 / worst 프레임 시간(ms), jank(16.7ms 초과) 수, build/raster 분해, 가시 마커 수를 기록
- 자동 벤치마크 러너가 규모 x 렌더 전략 x 줌 프리셋을 순회. 카메라는 고정 프리셋으로 재현하고, 조건 변경 후 4초 안정화 뒤 수집. 틱 간격(500ms)을 초과하는 프레임이 나오면 실패로 판정하고 조기 중단
- 렌더 전략: A(매 틱 전체 삭제 후 재생성), B(마커 캐시 + setPosition diff), C(B + viewport 컬링 + 1.5px 미만 이동 스킵 + 줌아웃 그리드 클러스터)

결과 1. 같은 1,000대, 렌더 방식 비교

| 모드 | avg (ms) | p95 (ms) | jank/윈도우 |
|------|------|------|------|
| A. 전체 재생성 | 24~31 | 210~270 | 스톨 반복 |
| B. 캐시+diff | 6.2~6.7 | 17.5~18.1 | 0~1 |
| C. 컬링+클러스터 | 12.1~12.6 | 14.4~15.1 | 0 |

A는 0.5초마다 화면이 1/4초씩 얼어붙는 수준으로, 1,000대에서 이미 실사용이 불가능합니다. A의 p95는 줌 레벨과 무관하게 동일했는데, 이는 비용이 화면에 보이는 수가 아니라 전체 차량 수에 종속된다는 직접 증거입니다.

결과 2. 규모 스케일링 (1,000 → 100,000대)

| 차량 수 | B p95 (ms) | C@z14 p95 (ms) | C 가시 마커 | C@z12 p95 (ms) | 클러스터 셀 |
|------|------|------|------|------|------|
| 1,000 | 17.5~18.1 | 14.4~15.1 | 15 | 27.4~27.6 | 31~32 |
| 5,000 | 5.8~16.2 | 11.6~12.9 | 58 | 25.3~27.0 | 35 |
| 10,000 | 8.9~18.4 | 11.7~13.3 | 118~122 | 23.7~26.9 | 35 |
| 100,000 | (측정 제외) | 10.2~12.8 | 1,120~1,143 | 12.4~18.7 | 35 |

- C는 100,000대에서도 p95 10~13ms, jank 0을 유지합니다. 줌 12에서는 10만 대 전부가 화면 범위에 들어오지만 클러스터 셀 35개로 압축됩니다. 셀 수는 화면 크기에 종속이라 1,000대든 100,000대든 35개로 고정입니다.
- 갱신 비용을 전체 차량 수가 아니라 화면에 보이는 수에 종속시킨다는 설계 목표가 수치로 확인됩니다.
- z12 윈도우 간 변동(12~27ms)은 지도 타일 렌더가 겹친 시뮬레이터 환경 특성으로, 규모 축과는 무관합니다(100,000대가 1,000대보다 빠르게 측정되기도 함).

## 가장 어려웠던 문제

문제 상황

규모를 10,000대로 올리자 프레임이 1.19초까지 늘어나며 사실상 정지했습니다. 5,000대까지는 멀쩡했기 때문에 처음에는 매 틱 10,000건씩 발생하는 setPosition 채널 호출이 원인이라고 판단했습니다.

시도한 해결 방법

프레임 계측기의 분해 수치를 다시 봤습니다. build와 raster는 각각 1~2ms뿐인데 프레임 전체는 1.19초였습니다. 렌더 파이프라인 안에서 쓰는 시간이 거의 없다는 뜻이라, 원인은 렌더가 아니라 그 바깥에 있었습니다. 추적해 보니 isolate가 매 틱 차량 10,000대를 `List<Map<String, dynamic>>`으로 직렬화해서 보내고 있었고, 이 구조 순회 직렬화가 메인 스레드를 잡아먹고 있었습니다.

최종 해결 방법

isolate 전송 프로토콜을 Float64List 버퍼(`[lat, lng, heading, speed] x N`)로 교체했습니다. 타입드 데이터는 구조 순회 없이 버퍼째 저비용으로 전달됩니다. 불변인 차량 ID 리스트는 매 틱 재생성하지 않고 한 번만 만들어 재사용했습니다. 같은 조건의 B 모드가 p95 9~18ms로 정상화되었고, 이 파이프라인 위에서 100,000대 측정까지 진행할 수 있었습니다.

이 과정에서 배운 것은 병목은 계측 없이 귀속하면 틀린다는 것, 그리고 병목은 한 곳에 고정된 것이 아니라 규모에 따라 렌더에서 데이터 파이프라인으로 이동한다는 것입니다.
